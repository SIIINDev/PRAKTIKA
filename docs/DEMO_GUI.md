# Демонстрация в GUI — сценарий для защиты

Пошаговый сценарий показа работающей системы через веб-интерфейс. Рассчитан на **5–7 минут**.
Каждый шаг привязан к ID требований, чтобы комиссия видела покрытие. Скриншоты работающих
экранов — в `.context/gui-demo/` (генерируются прогоном GUI).

> URL на стенде: интерфейс **http://localhost:8080**, Swagger **http://localhost:8000/docs**,
> Grafana **http://localhost:3001** (admin/admin), Prometheus **http://localhost:9091**.

## Перед показом (30 секунд)

```bash
cd ~/Documents/GitHub/PRAKTIKA
docker compose ps                 # все сервисы Up
curl -s localhost:8000/health     # status ok, все зависимости up
```
Открой в браузере вкладки заранее: `:8080`, `:8000/docs`, `:3001`. Не показывай пустую систему —
если индекс чист, заранее выполни `./init.sh` (загрузит 10 PDF-лекций).

## Сценарий (что говорить ↔ что нажимать)

### Шаг 1. Загрузка документов (FE-01, FE-02, FE-03, BE-01..05)
1. Открой **http://localhost:8080**, вкладка **«Документы»**.
2. Скажи: «Главная страница — зона drag-and-drop с множественной загрузкой».
3. **Перетащи** `tests/fixtures/lecture_databases.docx` и `tests/fixtures/lecture_algorithms.pdf`
   в область загрузки (или кликни и выбери оба).
4. Покажи **прогресс-бар и смену статусов**: `Загрузка… → Индексация… → Готово`.
   → Это FE-02 и весь backend-конвейер: приём файла (BE-01), валидация (BE-02), UUID (BE-03),
   извлечение текста pdfplumber/python-docx (BE-04), разбиение на чанки 1000/100 (BE-05).
5. Ниже — **список загруженных** с именем, датой, размером, числом фрагментов и статусом (FE-03).
6. Покажи документ со статусом **«Ошибка»** (битый PDF) — система корректно отрабатывает плохой ввод.

> Скриншот: `01-documents.png`.

### Шаг 2. Проверка валидации (BE-02) — по желанию, эффектно
1. Перетащи `tests/fixtures/notes.txt` (неверный тип).
2. Покажи, что файл отклонён (ошибка) — поддерживаются только PDF и DOCX, лимит 20 МБ.

### Шаг 3. Полнотекстовый поиск с подсветкой (FE-04..06, BE-06..09)
1. Вкладка **«Поиск»**.
2. Введи `нейронные сети`, нажми **Найти** (или **Enter** — покажи оба способа, FE-04).
3. Результаты — **карточки**: имя файла, № страницы, фрагмент, релевантность (FE-05).
4. Обрати внимание на **жёлтую подсветку** совпадений (FE-06).
5. Скажи: «Поиск идёт в Elasticsearch с русским анализатором — `multi_match` по тексту,
   ранжирование по релевантности» (BE-06, BE-08). Морфология: запрос «база данных» находит
   «базы данных».

> Скриншоты: `02-search-highlight.png`, `03-search-english-highlight.png`.

### Шаг 4. Пагинация (FE-07)
1. Введи частый запрос, например `model` (десятки результатов).
2. Прокрути вниз — **пагинация** `‹ 1 2 … 14 ›`, по 10 на страницу.
3. Нажми страницу **2** — выдача меняется.

> Скриншот: `04-pagination-page2.png`.

### Шаг 5. Пустой результат (FE-08)
1. Введи `zzzнетакого123`.
2. Покажи сообщение: **«По вашему запросу ничего не найдено. Попробуйте изменить формулировку»**.

> Скриншот: `05-empty-state.png`.

### Шаг 6. Адаптивность (FE-09)
1. Открой DevTools (F12) → переключатель устройства, поставь ширину **320px**.
2. Покажи, что вёрстка перестраивается в одну колонку, **нет горизонтальной прокрутки**.
3. Верни 1920px — раскладка снова широкая.

> Скриншот: `06-responsive-320.png`.

### Шаг 7. Документация API — Swagger (требование к API)
1. Открой **http://localhost:8000/docs**.
2. Покажи список эндпоинтов (OpenAPI 3.0): upload, search, documents, history, health.
3. Можно раскрыть `GET /api/v1/search`, нажать **Try it out**, ввести `q=алгоритм`, **Execute** —
   живой ответ с кодами 200/400.

> Скриншот: `07-swagger.png`.

### Шаг 8. Мониторинг — Grafana (DO-06)
1. Сначала «прогрей» метрики: на вкладке Поиск сделай 3–5 запросов.
2. Открой **http://localhost:3001** (admin/admin) → **Dashboards → «KB Search — API Metrics»**.
3. Покажи панели: число запросов к `/search` и время ответа (данные из Prometheus,
   цель `kb-backend` — UP на http://localhost:9091/targets).

## Что показать дополнительно (если спросят про тесты, DO/QA)

В терминале (это уже не GUI, но усиливает защиту):
```bash
# юнит-тесты + покрытие (QA-01)
cd backend && source .venv/bin/activate && pytest        # 27 passed, 64%
# E2E весь путь в браузере автоматически (QA-02)
cd ../tests/e2e && npx playwright test --reporter=line   # 1 passed
```
Отчёты: нагрузка — `tests/load/REPORT.md` (50 польз., 0 ошибок), Precision@3 — `tests/precision/REPORT.md` (hit@3 100%).
