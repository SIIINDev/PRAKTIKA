# Защита роли: Backend-инженер (BE)

Документ для подготовки к защите своей части проекта.

---

## Моя зона ответственности

> Разработка API, интеграция с Elasticsearch, бизнес-логика, работа с базами данных.

Я отвечал за весь серверный слой: REST API на FastAPI, конвейер обработки документов
(извлечение текста → чанкинг → индексация), полнотекстовый поиск через Elasticsearch,
кэширование в Redis и хранение метаданных в PostgreSQL.

## Ключевые технические решения

- **Асинхронный стек.** FastAPI + async SQLAlchemy (asyncpg) + async-клиенты ES и Redis —
  чтобы не блокировать event loop при I/O-операциях.
- **Фоновая индексация (`BackgroundTasks`).** Аплоад возвращает `202` сразу, тяжёлый разбор
  PDF идёт в фоне (`process_document`), статус документа обновляется в БД.
- **Разделение хранилищ.** PostgreSQL — метаданные и история; Elasticsearch — поисковый индекс.
  Каждое хранилище делает то, в чём оно сильно.
- **«Чистые» сервисы.** `chunker`, `validation`, `extractor` написаны как чистые функции без
  зависимости от HTTP/БД — их легко тестировать юнит-тестами.
- **Идемпотентность.** `ensure_index` создаёт индекс только если его нет; `init_db` создаёт
  таблицы безопасно. Старт приложения деградирует мягко, если зависимость недоступна.
- **Безопасность аплоада.** Двойная проверка: расширение И MIME; non-root пользователь в
  контейнере; ограничение размера на уровне приложения.

## Как продемонстрировать на защите

1. Открыть **Swagger**: http://localhost:8000/docs — показать все эндпоинты и схемы.
2. Через Swagger или интерфейс загрузить PDF/DOCX, показать ответ `202` с `id` и статусом.
3. Запросить `GET /documents/{id}` несколько раз — показать переход статуса до `done` и `chunk_count`.
4. Выполнить `GET /api/v1/search?q=нейронные сети` — показать `score`, `highlight`, `page`, `total`, `took_ms`.
5. Повторить тот же запрос — показать `cached: true` (ответ из Redis).
6. Показать обработку ошибок: запрос с пустым `q` → 400; аплоад `.txt` → 400; битый PDF → статус `error`.
7. Показать `/health` — все зависимости `up`.
8. Запустить тесты: `cd backend && pytest` — 25 тестов, покрытие 62%.

Полезные команды:

```bash
curl -s http://localhost:8000/health
curl -s -F "file=@tests/fixtures/lecture_algorithms.pdf;type=application/pdf" \
     http://localhost:8000/api/v1/documents/upload
curl -s "http://localhost:8000/api/v1/search?q=бинарный%20поиск"
```

## Возможные вопросы комиссии и ответы

**1. Почему чанки именно 1000 символов с перекрытием 100?**
1000 символов — компромисс между точностью (мелкие фрагменты лучше локализуют совпадение)
и достаточным контекстом для подсветки. Перекрытие 100 символов гарантирует, что фраза на
границе двух чанков не «разорвётся» и будет найдена целиком. Параметры вынесены в конфиг
(`chunk_size`, `chunk_overlap`), их можно менять без правки кода.

**2. Как работает `multi_match`?**
Запрос ищется одновременно по нескольким полям — у нас `text` (основное) и `file_name`.
ES прогоняет запрос через русский анализатор, ищет совпадения в каждом поле и берёт лучший
скоринг. Это удобнее одиночного `match`, потому что релевантным считается и попадание в имя файла.

**3. Зачем русский анализатор и что он делает?**
Встроенный анализатор `russian` выполняет токенизацию, нижний регистр, стоп-слова и стемминг.
Благодаря стеммингу запрос «нейронные сети» находит «нейронных сетей», «нейронной сети» и т. п. —
без него совпадали бы только точные словоформы.

**4. Зачем Redis и почему TTL 5 минут?**
Redis кэширует результаты частых одинаковых запросов: ключ `search:{q}:{page}:{size}`, ответ
сериализуется в JSON. Это снижает нагрузку на ES и ускоряет повторный поиск. TTL 5 минут (по
заданию) — баланс между актуальностью (новые документы быстро попадут в выдачу) и эффективностью.

**5. Что будет, если Redis или Elasticsearch упадёт?**
Кэш реализован с мягкой деградацией: ошибки `get`/`set` логируются, но запрос всё равно идёт в ES.
Если упадёт ES — поиск вернёт `503` с понятным сообщением, а не абстрактную 500. Старт приложения
тоже не падает: `/health` покажет, какая зависимость недоступна.

**6. Почему аплоад возвращает 202, а не 200?**
`202 Accepted` означает «принято в обработку». Извлечение текста из PDF может занять время,
поэтому мы не держим HTTP-соединение, а запускаем фоновую задачу и сразу отдаём `id`. Клиент
отслеживает прогресс опросом `GET /documents/{id}`.

**7. Как обеспечивается консистентность при удалении документа?**
`DELETE /documents/{id}` удаляет чанки из ES (`delete_by_query` по `document_id`), затем файл с
диска, затем строку в БД. Ошибка удаления чанков ES логируется, но не блокирует удаление строки,
чтобы не оставить «висящую» запись.

**8. Как вы извлекаете номер страницы для DOCX?**
В DOCX нет жёсткой пагинации, как в PDF, поэтому весь текст документа считается «страницей 1»
(включая текст таблиц). Для PDF pdfplumber даёт реальную постраничную разбивку.

**9. Как тестируется backend без поднятия ES/Redis/PostgreSQL?**
Чистые функции (`chunker`, `validation`, `extractor`) тестируются напрямую. API-тесты
(`test_api.py`) поднимают приложение через ASGITransport и подменяют (monkeypatch) внешние
зависимости заглушками — суите не нужны живые сервисы.

**10. Что хранится в PostgreSQL, а что в Elasticsearch?**
В PostgreSQL — метаданные документа (имя, размер, статус, число чанков, пути, даты) и история
поиска. В Elasticsearch — сами чанки текста с метаданными для поиска и подсветки. Состояние и
жизненный цикл — в реляционной БД, индекс и релевантность — в поисковом движке.
