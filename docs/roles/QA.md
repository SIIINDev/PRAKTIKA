## Ключевые технические решения

- **Юнит-тесты без живых сервисов.** Чистая логика (`chunker`, `validation`, `extractor`)
  тестируется напрямую. API-тесты поднимают приложение через ASGITransport и подменяют ES,
  Redis, PostgreSQL заглушками (monkeypatch) — суите не нужны контейнеры, она быстрая и
  детерминированная, годится для CI.
- **Покрытие в CI.** `pytest` с `--cov=app --cov-report=term-missing` (конфиг в `pyproject.toml`):
  покрытие считается автоматически, видно непокрытые строки. Текущее — 62% при пороге 50%.
- **E2E на реальном стеке.** Playwright ходит в реально запущенный фронт (:8080) и проверяет
  сквозной сценарий пользователя через `data-testid`-селекторы.
- **Осмысленные фикстуры.** Каждый тестовый файл проверяет конкретный путь: happy-path,
  постраничность PDF, пустой файл, битый файл, неверный формат. Бинарники лежат только в
  `tests/fixtures` (разрешено заданием) и воспроизводимо генерируются скриптом.
- **Объективная оценка поиска.** Precision@3 — измеримая метрика релевантности вместо
  «на глаз»: для эталонных запросов проверяем, попал ли нужный документ в топ-3.

## Как продемонстрировать на защите

1. **Юнит-тесты + покрытие:**
   ```bash
   cd backend && pytest
   ```
   Показать: 25 тестов проходят, покрытие 62%, ruff без замечаний.
2. **E2E (на поднятом стеке):**
   ```bash
   cd tests/e2e && npx playwright test
   ```
   Показать сценарий: загрузка docx → статус «Готово» → поиск → карточки с `<mark>` → пустое
   состояние. Итог — в `tests/e2e/REPORT.md`.
3. **Фикстуры:** показать `tests/fixtures/README.md` и продемонстрировать поведение системы на
   битом PDF (статус `error`) и на `.txt` (HTTP 400).
4. **Нагрузка (Locust):** запустить сценарий на 50 пользователей, показать `tests/load/REPORT.md`
   (время отклика).
5. **Precision@3:** показать таблицу результатов в `tests/precision/REPORT.md`.
6. **Документация:** показать `docs/USER_GUIDE.md`.

## Возможные вопросы комиссии и ответы

**1. Как вы считаете покрытие кода и какое оно?**
Через `pytest --cov=app` (плагин coverage, настроен в `pyproject.toml`). Coverage инструментирует
код и считает долю выполненных строк. Сейчас 62% при требуемом пороге 50%. Покрыты ключевые
модули: валидация, чанкинг, извлечение текста, обработка ошибок API.

**2. Почему именно эти модули покрыты тестами?**
Приоритет — на бизнес-логике с наибольшим риском ошибок и без внешних зависимостей: парсинг
документов, чанкинг (границы окна, перекрытие), валидация (форматы, размер), коды ответов API.
Это требование QA-01 (парсинг и валидация) и самые ошибкоопасные места.

**3. Что такое Precision@3 и как вы её измеряете?**
Precision@3 — доля запросов, для которых релевантный (эталонный) документ оказался в первых
трёх результатах выдачи. Берём 10 заранее заданных запросов с известным «правильным» документом,
выполняем поиск, проверяем топ-3. Метрика объективно оценивает качество ранжирования. Результаты —
в `tests/precision/REPORT.md`.

**4. Как устроено нагрузочное тестирование?**
Locust имитирует 50 одновременных пользователей, выполняющих поисковые запросы к `/search`.
Собираются метрики времени отклика (среднее, перцентили) и RPS. Отчёт — в `tests/load/REPORT.md`.
Дополнительно нагрузку видно в Grafana-дашборде в реальном времени.

**5. Чем E2E-тесты отличаются от юнит-тестов?**
Юнит-тесты проверяют отдельные функции в изоляции (без сервисов). E2E проверяет систему целиком
глазами пользователя: реальный браузер открывает фронт, загружает файл, ждёт индексацию, ищет и
проверяет подсветку. Это требование QA-02 — критический сквозной сценарий.

**6. Какие тестовые документы вы подготовили и зачем?**
`lecture_databases.docx` и `lecture_algorithms.pdf` — корректные (happy-path и постраничность),
`empty.pdf` — пустой (должен дать статус `error`), `broken.pdf` — битый (тоже `error`),
`notes.txt` — неверный формат (HTTP 400). Покрывают и позитивные, и негативные сценарии (QA-03).

**7. Почему юнит-тесты не требуют запущенных ES/Redis/PostgreSQL?**
Чистые функции тестируются напрямую. Для API-тестов внешние зависимости заменяются заглушками
через monkeypatch, а приложение поднимается in-process через ASGITransport. Это делает тесты
быстрыми, стабильными и пригодными для запуска в CI без Docker.

**8. Как вы убедились, что система корректно обрабатывает ошибки?**
И юнит-тестами (валидация отклоняет неверные форматы/размеры; экстрактор падает на пустых/битых
файлах), и вручную/в E2E: `.txt` → 400, пустой `q` → 400, битый PDF → статус `error` с
сообщением. Проверялись именно негативные сценарии, а не только happy-path.

**9. Как контролируется качество кода помимо тестов?**
Линтер ruff (PEP 8 и набор правил) в CI на каждый push; типизация фронтенда `tsc --noEmit`;
требование документировать публичные функции (docstrings) выполнено в backend. Всё это входит в
пайплайн CI.

**10. Что ещё можно улучшить в тестировании?**
Поднять покрытие до 70%+, добавить интеграционные тесты с реальным ES (в отдельном job CI с
сервисами), расширить E2E на сценарии множественной загрузки и пагинации, автоматизировать
прогон Precision@3 в CI. Это зафиксировано как направления развития.
