Документ для подготовки к защите своей части проекта.

---

## Моя зона ответственности

> Разработка пользовательского интерфейса, вёрстка, клиент-серверное взаимодействие.

Я отвечал за SPA на React + TypeScript: интерфейс загрузки документов (drag-and-drop,
прогресс, статусы), интерфейс поиска (поле, карточки, подсветка, пагинация), клиент REST API
и адаптивную вёрстку.

## Ключевые технические решения

- **TypeScript-контракт с backend.** Типы ответов API (`Document`, `SearchResponse` и др.)
  описаны в `services/api.ts` строго по `docs/API_CONTRACT.md`, что ловит несоответствия на
  этапе компиляции (`tsc --noEmit`).
- **Реальный прогресс загрузки.** Для аплоада используется `XMLHttpRequest` (а не `fetch`),
  потому что только он даёт событие `upload.onprogress` с реальным процентом передачи файла.
- **Опрос статуса (polling).** После приёма файла (`202`) фронт опрашивает `GET /documents/{id}`
  каждые 1.5 с до статуса `done`/`error`, обновляя прогресс-бар и список. Таймеры аккуратно
  очищаются при размонтировании, чтобы не было утечек.
- **Безопасная подсветка без `dangerouslySetInnerHTML`.** Компонент `Highlight` сам токенизирует
  строку, считая разметкой только литералы `<em>`/`</em>`, остальное рендерится как текст
  (React экранирует). Это исключает XSS из содержимого документов.
- **Отмена устаревших запросов.** При новом поиске предыдущий запрос отменяется через
  `AbortController` — нет «гонок» и мигания старых результатов.
- **Адаптивность без фиксированных ширин.** Грид и media-запросы в `styles.css`; проверено,
  что на 320px нет горизонтального переполнения.

## Как продемонстрировать на защите

1. Открыть http://localhost:8080, показать две вкладки: **Документы** и **Поиск**.
2. На вкладке **Документы** перетащить несколько PDF/DOCX в drop-зону.
3. Показать прогресс-бар и смену статусов: Загрузка… → Индексация… → Готово.
4. Показать список документов: имя, дата, размер, число фрагментов, статус.
5. Перейти на **Поиск**, ввести запрос и нажать **Enter** (показать, что работает и Enter, и кнопка).
6. Показать карточки: имя файла, «стр. N», фрагмент, оценку, **жёлтую подсветку**.
7. Полистать страницы (пагинация по 10), показать прокрутку к началу.
8. Ввести заведомо отсутствующий запрос — показать сообщение «ничего не найдено».
9. Нажать на чип в истории — показать повтор запроса.
10. Уменьшить окно браузера до ~320px — показать адаптивность.
11. (По желанию) `cd frontend && npm run typecheck && npm run build` — типизация и сборка проходят.

## Возможные вопросы комиссии и ответы

**1. Как вы безопасно подсвечиваете совпадения? Нет ли XSS?**
Сервер присылает фрагмент с тегами `<em>…</em>`. Я НЕ вставляю это как HTML. Компонент
`Highlight` посимвольно проходит строку, распознаёт только литералы `<em>`/`</em>` как
границы выделения, а весь остальной текст кладёт в обычные React-узлы (React их экранирует).
Любые «левые» теги или скрипты из документа отрендерятся как безобидный текст. Подсветка —
это `<mark>` с жёлтым фоном из CSS.

**2. Почему для загрузки XHR, а не fetch?**
`fetch` не умеет отдавать прогресс отправки тела запроса. Чтобы показать реальный процент
загрузки файла, я использую `XMLHttpRequest` и его событие `upload.onprogress`.

**3. Как реализован прогресс/статусы после загрузки?**
Загрузка файла — это только первый этап. После ответа `202` сервер индексирует документ в фоне.
Фронт опрашивает `GET /documents/{id}` раз в 1.5 с и обновляет статус, пока не придёт
`done` или `error`. Для индексации показывается «бегущий» (indeterminate) прогресс-бар.

**4. Как сделана адаптивность под 320–1920px?**
Без жёстких ширин: основная сетка на CSS Grid с media-запросами, контейнеры тянутся по ширине,
длинные строки переносятся. Проверено инструментом ui-test: на 320px нет горизонтального скролла.

**5. Поиск запускается по Enter и по кнопке — как?**
Поле и кнопка обёрнуты в `<form>` с `onSubmit`. Нажатие Enter в поле и клик по кнопке `submit`
дают один и тот же обработчик. Кнопка заблокирована, пока запрос пустой или идёт загрузка.

**6. Что происходит при быстром вводе нескольких запросов подряд?**
Каждый новый поиск отменяет предыдущий через `AbortController`. Прерванный запрос игнорируется
(ошибка `AbortError` не показывается), отображается результат только последнего запроса.

**7. Как обрабатываются ошибки сети и сервера?**
Клиент API разбирает тело ошибки (поле `detail`) и пробрасывает понятное сообщение.
В интерфейсе ошибки показываются в баннере (`role="alert"`), а у задачи загрузки — текст причины.

**8. Откуда берётся метка «из кэша»?**
Сервер в ответе на поиск возвращает поле `cached`. Если оно `true`, рядом со сводкой
показывается метка «из кэша». Фронт ничего сам не кэширует — это серверный Redis.

**9. Как вы заботились о доступности (accessibility)?**
ARIA-метки на drop-зоне, прогресс-баре (`role="progressbar"`), навигации и пустых состояниях;
скрытые подписи для поля поиска; управление с клавиатуры (drop-зона активируется Enter/Space).
Проверено axe-core — 0 нарушений.

**10. Почему React + Vite + TypeScript?**
React — по заданию (React или Vue). TypeScript даёт типобезопасный контракт с API и ловит
ошибки на компиляции. Vite — быстрый dev-сервер с проксированием `/api` и компактная
production-сборка, которую отдаёт nginx.
