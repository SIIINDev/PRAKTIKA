# План защитной презентации

Интеллектуальная поисковая система по внутренней базе знаний университета.
Учебная практика, направление 09.03.04. Команда из 4 человек (BE / FE / DO / QA).

Рекомендуемый объём — 7 слайдов. Длительность защиты — ~7–10 минут + ответы на вопросы.

---

## Слайд 1. Проблема и цель

**Заголовок:** Поиск по внутренней базе знаний университета

- **Проблема:** материалы (лекции, методички, документы) разрознены; найти нужный фрагмент
  по ключевым словам сложно, обычный поиск по имени файла не помогает.
- **Цель:** веб-приложение, которое индексирует загруженные PDF/DOCX и даёт быстрый
  полнотекстовый поиск с ранжированием и подсветкой совпадений.
- **Что сделали:** полнофункциональный продукт на современном стеке, запускается одной командой.
- **Ключевые цифры:** 26 требований выполнено; 25 юнит-тестов, покрытие 62%; запуск
  `docker compose up` одной командой.

## Слайд 2. Архитектура

**Заголовок:** Архитектура и технологический стек

- Схема компонентов: React SPA (nginx) → FastAPI → PostgreSQL / Elasticsearch / Redis;
  Prometheus + Grafana для мониторинга (вставить ASCII-схему из `README.md`).
- **Разделение ответственности:** PostgreSQL — метаданные и история; Elasticsearch — индекс
  и релевантность; Redis — кэш.
- **Стек:** Python 3.12 + FastAPI, React 18 + TypeScript, Elasticsearch 8.13 (russian analyzer),
  PostgreSQL 16, Redis 7, Docker Compose, GitHub Actions.
- Браузер общается с одним origin (nginx проксирует `/api`) — без проблем с CORS.

## Слайд 3. Демо-сценарий

**Заголовок:** Как это работает (живая демонстрация)

1. Открываем http://localhost:8080.
2. Перетаскиваем PDF/DOCX → статусы Загрузка… → Индексация… → Готово.
3. Идём в «Поиск», вводим запрос (например, «нейронные сети»), жмём Enter.
4. Видим карточки: имя файла, страница, фрагмент, **жёлтая подсветка**, оценка релевантности.
5. Повторяем запрос → метка «из кэша» (Redis).
6. Листаем страницы (по 10); вводим несуществующий запрос → «ничего не найдено».
7. Показываем Swagger `/docs` и `/health` (все зависимости `up`).

> Подсказка: заранее загрузить документы через `./init.sh`, чтобы поиск сразу был наполнен.

## Слайд 4. Технологии и ключевые решения

**Заголовок:** Технические решения и почему так

- **Чанкинг 1000/100** — баланс точности и контекста; перекрытие не «рвёт» фразы на границе.
- **Русский анализатор ES** — стемминг находит разные словоформы («сети» ↔ «сетей»).
- **Фоновая индексация + статусы** — аплоад отвечает мгновенно (202), прогресс через polling.
- **Кэш Redis с мягкой деградацией** — ускоряет повторные запросы, отказ кэша не ломает поиск.
- **Безопасная подсветка** — без `dangerouslySetInnerHTML`, защита от XSS из содержимого документов.
- **Healthcheck + depends_on** — `app` стартует только после готовности БД/ES/Redis.

## Слайд 5. Тестирование и метрики

**Заголовок:** Контроль качества

- **Юнит-тесты (QA-01):** 25 тестов pytest, покрытие 62% (>50%), ruff без замечаний.
- **E2E (QA-02):** Playwright — сквозной сценарий загрузка → индексация → поиск → подсветка
  → пустая выдача (итог в `tests/e2e/REPORT.md`).
- **Тестовые данные (QA-03):** корректные / пустой / битый / неверный формат.
- **Нагрузка (QA-04):** Locust, 50 пользователей по /search (итог в `tests/load/REPORT.md`).
- **Качество поиска (QA-05):** Precision@3 по 10 запросам (итог в `tests/precision/REPORT.md`).
- **Мониторинг:** Grafana-дашборд — rps поиска, латентность p95/p50 (показать вживую).

## Слайд 6. DevOps и репозиторий

**Заголовок:** Сборка, запуск, CI/CD

- **Одна команда:** `cp .env.example .env && docker compose up --build`.
- **CI (GitHub Actions):** при push в main — ruff + pytest, tsc + build, затем сборка образов.
- **Конфигурация через `.env`** — секретов в коде нет; порты настраиваются.
- **init.sh** — наполнение базы 10 PDF-лекциями из открытого доступа.
- Структура репозитория соответствует шаблону задания.
